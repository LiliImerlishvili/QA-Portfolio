# 🐞 BUG-007 – Speaker-to-Text Mapping Is Inconsistent in Multi-Speaker Mode

---

## 📌 Summary
In **Multi-Speaker** mode, the system does not preserve correct speaker-to-text mapping.  
Generated audio alternates voices incorrectly within a single speaker input field, making dialogue generation unpredictable.

---

## 🖥 Environment

| Parameter | Value |
|---------|------|
| Platform | Web |
| Device | iMac |
| Browser | Google Chrome 143.0.7499.193 (Official Build, arm64) |
| OS | iOS Tahoe 26.2 |
| Reproducibility | Always (100%) |

---

## 🏷 Labels

- Web  
- AI Audio  
- Multi-Speaker  
- High Priority  

---

## 📝 Description

In **Multi-Speaker mode**, the system fails to maintain a strict mapping between **speaker input fields** and their selected voices.

When multiple sentences are entered into the input field assigned to a specific speaker, the generated audio alternates voices between different speakers within the same field.

### Example:
- In **Speaker 1 (Charon)** input field, the first sentence is read by Charon, but the next sentence is read by **Speaker 2 (Zephyr)**.
- The same behavior occurs in **Speaker 2 (Zephyr)** input field, where sentences are also read alternately by both speakers.

As a result, the speaker assigned to an input field does not consistently read the content of that field, making multi-speaker dialogue generation confusing and unreliable for end users.

---

## 🔁 Preconditions

- User is authenticated
- User has an active **Silver subscription**
- User has sufficient credits available
- User is on the **Chat Box** page

---

## 🧪 Steps to Reproduce

1. Click on **AI Audio**
2. Select **Multi-Speaker** mode
3. Enter a valid prompt
4. Select **Speaker 1 – Charon**
5. Select **Speaker 2 – Zephyr**
6. Fill in the speaker input fields with **multiple sentences**
7. Click the **Generate** button
8. Listen to the generated audio output

---

## ❌ Actual Result

- Voices alternate between **Charon** and **Zephyr** within a single speaker input field
- Text entered for one speaker is partially read by the other speaker
- Speaker-to-text mapping is not respected

---

## ✅ Expected Result

- Each speaker input field is **strictly bound** to its selected speaker
- All sentences entered in **Speaker 1 (Charon)** field are read only by Charon
- All sentences entered in **Speaker 2 (Zephyr)** field are read only by Zephyr
- Dialogue generation is predictable and consistent

---

## 📉 Impact

- Multi-speaker audio generation behaves unpredictably
- Users cannot control which voice reads their text
- Feature appears broken during demos and real usage
- Reduces trust in audio generation quality

---

## 🔄 Frequency

---

## 📝 Notes

- Issue occurs regardless of language, punctuation, or voice selection
- Fully reproducible across multiple attempts
- Not related to subscription level or credit balance

---

## 💡 Recommendation

Enforce a **strict binding** between speaker input fields and their selected voices.  
Ensure that all text entered for a specific speaker is read exclusively by that speaker.

This would prevent voice mixing within a single input field and make multi-speaker behavior predictable and reliable.
