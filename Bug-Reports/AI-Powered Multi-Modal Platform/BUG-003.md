# 🐞 BUG-003 – Multi-Speaker Mode Does Not Preserve Speaker-to-Text Mapping

---

## 📌 Summary
In **Multi-Speaker** mode, the system does not consistently preserve speaker-to-text mapping. Generated audio alternates voices between different speakers within the same input field, resulting in unpredictable and incorrect dialogue output.

---

## 🖥 Environment

| Parameter | Value |
|---------|-------|
| Platform | Web |
| Browser | Google Chrome 145.0.7499.195 (Official Build, arm64) |
| OS | iOS Tahoe 26.2 |
| Frequency | Always (100%) |

---

## 🏷 Labels

- Web  
- AI Audio  
- Multi-Speaker  

---

## 📝 Description

In **Multi-Speaker** mode, the system fails to bind a specific input field to its selected speaker.

When multiple sentences are entered into the input field assigned to a specific speaker, the generated audio alternates voices between different speakers **within the same input field**.

### Example:
- In **Speaker 1 (Charon)** input field:
  - First sentence is read by **Charon**
  - Next sentence is read by **Speaker 2 (Zephyr)**
- The same alternating behavior occurs in **Speaker 2 (Zephyr)** input field

As a result, speaker-to-text mapping is not respected, making multi-speaker dialogue generation unpredictable and confusing for users.

---

## 🔁 Preconditions

- User is authenticated
- User has an active **Silver subscription**
- User has sufficient credits available
- User is on the **Chat Box** page

---

## 🧪 Steps to Reproduce

1. Click on **AI Audio**
2. Select **Multi-Speaker** mode
3. Enter a valid prompt
4. Select **Speaker 1 – Charon**
5. Select **Speaker 2 – Zephyr**
6. Enter multiple sentences in the speaker input fields
7. Click **Generate**
8. Listen to the generated audio

---

## ❌ Actual Result

- Voices alternate between **Charon** and **Zephyr** within a single speaker input field
- Text entered for one speaker is partially read by the other
- Speaker-to-text mapping is not respected

---

## ✅ Expected Result

- Each speaker input field is strictly bound to its selected speaker
- All sentences in **Speaker 1 (Charon)** field are read only by Charon
- All sentences in **Speaker 2 (Zephyr)** field are read only by Zephyr
- Dialogue generation is predictable and consistent

---

## 📉 Impact

- Multi-speaker audio generation behaves unpredictably
- Users cannot control which voice reads their text
- Feature appears broken during demos and real usage
- Reduces trust in audio generation quality

---

## 🔄 Frequency

- Always (100%)

## ⚠ Severity

- High

---

## 🚦 Priority

- Highest

---


## 📝 Notes

This issue occurs regardless of:
- Language
- Punctuation
- Voice selection

The issue is consistently reproducible.

---

## 💡 Recommendation

Enforce a strict binding between speaker input fields and their selected voices, ensuring that all text entered for a speaker is read only by that speaker.

This will prevent voice mixing within a single input field and make multi-speaker behavior predictable and reliable.
